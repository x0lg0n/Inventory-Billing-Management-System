## Becoming a Maintainer

We are always looking for experienced contributors to help maintain this project. If you're interested in becoming a maintainer, here's what we look for:

- **Consistent, high-quality contributions** to the codebase
- **Understanding** of the project's architecture and goals
- **Positive interactions** with the community
- **Reliability** in reviewing pull requests and triaging issues
- **Commitment** to the project's future

### Process

1. Demonstrate commitment through regular contributions
2. Nominate yourself or be nominated by existing maintainers
3. Discussion among current maintainers
4. Consensus or vote to invite as maintainer
5. Acceptance and addition to the maintainers list

## Maintainer Responsibilities

All maintainers commit to:

- **Reviewing pull requests** promptly and constructively
- **Triaging issues** and labeling them appropriately
- **Ensuring code quality** and adherence to project standards
- **Maintaining documentation** accuracy
- **Supporting contributors** with guidance and mentorship
- **Acting in the best interests** of the project and community
- **Following the Code of Conduct** and promoting an inclusive environment

## Communication

### Response Times

While we are volunteers, we aim to:

- Acknowledge issues/PRs within **3 business days**
- Provide feedback on PRs within **7 business days**
- Triage reported bugs within **2 weeks**

### Channels

- **GitHub Issues & Discussions**: For public discussions and feature requests
- **Pull Requests**: Code review and feedback
- **Email**: `security@x0lg0n.dev` for security issues

## Decision Making

The project follows a consensus-based decision model. See [GOVERNANCE.md](GOVERNANCE.md) for detailed information about how decisions are made, voting procedures, and conflict resolution.

## Inactivity Policy

Maintainers with **no activity for 6+ months** and no communication may be marked as inactive. This helps keep the team engaged and responsible.

If you need to take a break, please let us know! We understand life happens and we're flexible.

## Code Review Guidelines

All maintainers should:

1. **Be respectful** and constructive in feedback
2. **Explain the reasoning** behind requested changes
3. **Acknowledge good work** and contributions
4. **Test the changes** when possible
5. **Check for consistency** with project standards
6. **Ensure documentation** is updated if needed
7. **Merge PRs** that pass review and CI checks
